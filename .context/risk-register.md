# 风险台账

| 风险 | 触发条件 | 影响 | 缓解策略 |
| --- | --- | --- | --- |
| Windows SmartScreen 拦截 | 新 EXE 未建立信誉或签名不足 | 用户不敢运行、售后成本升高 | 在交付文档中提供明确说明；中期评估代码签名；保持版本与来源可验证 |
| 杀软误报或隔离 `PyInstaller` 产物 | 打包产物包含 Qt/Node/runtime 目录，触发启发式误报 | 无法启动、用户误判为木马 | 建立白名单说明、保留产物校验值、减少可疑行为、避免高风险默认技能 |
| 微信等渠道风控 | 官方或社区接入方式被限制、设备迁移、扫码重绑 | 渠道无法持续使用、售后频率升高 | 将渠道设计为可插拔能力；在帮助页中明确风险边界；默认不承诺不可控渠道的长期稳定性 |
| 上游 OpenClaw breaking changes | 上游 API、目录、启动参数或插件机制变化 | 真实 runtime adapter 失效或升级失败 | 锁定上游版本快照；建立版本兼容矩阵；升级前做回归验证 |
| U 盘损坏或文件系统异常 | 长期插拔、异常断电、低质量介质 | 配置丢失、运行时损坏、售后复杂度增加 | 配置快照与备份目录；明确建议高质量介质；日志与缓存写入 `%TEMP%` 降低 U 盘写入压力 |
| Provider API 变化或 Key 失效 | Provider 改协议、限流、欠费或 Key 撤销 | 向导测试失败、用户无法对话 | Provider 模板可热更新；在 UI 中明确错误归因；提供离线模式与重新配置入口 |
| 真实 runtime 与桌面壳层耦合过深 | 为快速接入而让 UI 直接依赖运行时内部细节 | 后续升级成本高、Mock/Real 切换返工 | 严守 `RuntimeAdapter` 边界；通过视图模型承接状态，不把内部实现泄漏到界面层 |
| 更新覆盖失败 | 下载包不完整、写入失败、文件被占用 | 启动失败、用户数据受损 | 更新前自动备份 `runtime/`；首启失败自动回滚；永不覆盖 `state/` |
| 诊断包泄露敏感信息 | 日志、配置、token 未脱敏就导出 | 用户密钥泄露、商业与安全风险 | 严格执行脱敏规则；诊断包默认不包含消息正文；导出前再次校验敏感字段 |
| Windows 长路径限制 | `runtime/openclaw/` 包含深层 `dist/extensions/*/node_modules`，且交付目录或临时打包目录过深 | 复制、打包或更新时文件丢失，导致 runtime 启动失败 | U 盘交付目录保持短路径；打包脚本避免深层临时目录；必要时评估依赖裁剪或长路径策略 |
