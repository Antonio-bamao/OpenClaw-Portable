# OpenClaw Portable 总体计划

## 项目目标

- 交付一个面向 Windows 10/11 x64 的便携式 OpenClaw 桌面壳层，适配 U 盘交付场景。
- 让非技术用户通过桌面启动器完成配置、启动、停止、打开 WebUI、查看状态与诊断，不依赖命令行。
- 从 Day 1 就按最终产品结构组织项目，避免后续接入真实 OpenClaw、渠道插件和售后系统时推倒重来。
- 建立 `.context/` 工程指挥舱，使计划、状态、日志、Bug 根因和关键决策可持续追踪。

## 当前现实基线

- 已完成 Phase 1 开发版 MVP 的基础壳层：
  - `PySide6` 原生桌面启动器与自定义主题
  - 首次引导向导与主控制面板
  - `PortablePaths`、`LauncherConfigStore`、`PortResolver`、`LauncherController`
  - `MockRuntimeAdapter` 与 Node mock runtime
  - Provider JSON 模板
  - `PyInstaller onedir` 烟雾验证
- 尚未完成真实 OpenClaw runtime 封装、渠道接入、更新回滚、诊断包与售后闭环。

## 阶段与里程碑

### Phase 1｜桌面启动器 MVP 基线

目标：
- 建立可运行、可打包、可演示的便携版壳层。
- 完成首次向导、主控制台、配置落盘、端口冲突处理、Node mock runtime 联调。

当前状态：
- 已基本完成并通过基础测试与打包烟雾验证。

剩余缺口：
- 完善离线帮助页与快速说明的一致性
- 完善诊断导出与重置入口
- 补足 `.context` 指挥舱并将后续研发切换为 context-first 工作流

验收标准：
- 源码态可运行
- 首次向导与主控制台可走通
- `state/openclaw.json` 与 `state/.env` 分层正确
- mock runtime 可启动、停止、重启、健康检查
- `PyInstaller onedir` 可成功产出桌面壳层

### Phase 2｜真实 OpenClaw Runtime 封装

目标：
- 将当前 mock runtime 替换为真实 OpenClaw runtime 适配层。
- 锁定 Node 版本、OpenClaw 版本、启动脚本、日志与缓存路径规则。

核心工作：
- 在 `runtime/` 中引入真实 Node 运行时与 OpenClaw 程序目录
- 新增真实 `RuntimeAdapter`，保持与现有 controller/UI 接口兼容
- 统一健康检查、日志采集、异常退出检测与端口重绑定反馈
- 建立回归测试，防止 mock → real 切换时界面层返工

验收标准：
- 双击启动器可拉起真实 OpenClaw runtime
- WebUI 地址能正确打开且健康检查稳定
- 日志、缓存、配置路径全部符合便携目录规范
- 真实 runtime 接入不要求重写主界面或配置存储逻辑

依赖：
- OpenClaw 上游版本优先锁定 `v2026.4.8`
- Node 版本策略调整为 Node 24 优先，Node 22.14+ 仅作为兼容备选
- 默认交付模式为原生 Windows 便携运行时，WSL2 仅作为高级用户或排障 fallback

### Phase 3｜渠道接入与管理面板

目标：
- 补齐微信、飞书、QQ、企微的接入引导、状态展示和基础管理能力。

核心工作：
- 设计渠道插件目录与状态存储规范
- 为每个渠道提供“启用 / 配置 / 连接 / 断开 / 删除”入口
- 内置帮助页、风险提示与设备迁移提示
- 设计登录态失效、换电脑重绑、受限权限下的提示文案

验收标准：
- 启动器或 WebUI 中能看到渠道状态
- 每个渠道都有最小接入闭环与帮助文档
- 渠道风险提示与用户决策点明确

依赖：
- 真实 runtime 能加载或托管插件
- 需要确认各渠道官方或社区方案的可行性与合规边界

### Phase 4｜售后、更新与商业化交付能力

目标：
- 建立面向真实出售场景的诊断、更新、回滚、恢复出厂与风险缓解机制。

核心工作：
- 一键诊断包导出与脱敏
- 更新检查、下载、覆盖与自动回滚
- 恢复出厂配置与备份恢复
- SmartScreen、杀软误报、U 盘损坏、上游 breaking changes 的应对方案
- README、免责声明、售后流程与交付清单

验收标准：
- 用户能在不依赖开发者的情况下完成基础自助排障
- 更新失败时可自动回滚
- 诊断包可直接用于售后定位问题
- 交付介质与文档达到可销售状态

## 横向工程原则

- 架构优先于短期拼功能，所有 Phase 1 代码都要可平滑演进到真实 runtime。
- 先落计划和日志，再做实施，避免“做一步想一步”。
- UI 走“可信、克制、现代 Windows 工具感”，而不是拼贴式 Demo 风格。
- Bug 处理必须记录根因与预防，不允许只修症状。

## 完成定义

项目可视为进入可销售准备阶段，至少满足以下条件：

- 真实 runtime 已接入并稳定运行
- 首次引导、主控制面板、渠道管理、诊断、更新、恢复出厂全部闭环
- 日志、缓存、配置、渠道状态路径全部满足便携版要求
- 核心测试、打包验证、回滚验证与诊断导出验证通过
- 风险台账中高风险项均有明确缓解策略或交付声明
