# 本地更新包合法性校验设计

## 背景

当前项目已经具备“导入本地更新包 -> 自动备份旧分发内容 -> 中途失败自动回滚”的最小闭环，但“导入更新包”入口仍缺少真正的合法性校验。现在只要求源目录不是当前目录且存在 `version.json`，还不能拦住空包、残包、同版本包或旧版本包。

项目同时已经具备“恢复更新备份”入口，因此“导入更新包”不需要承担降级或回退职责，应该明确收敛为面向小白用户的“升级”入口。

## 目标

为“导入更新包”补一层严格、面向非技术用户的合法性校验：

- 坏包、空包、残包直接拦截
- 同版本包直接拦截
- 旧版本包直接拦截，并明确提示改用“恢复更新备份”
- 只有严格高于当前版本的包才允许导入

## 非目标

- 本轮不做数字签名、哈希校验或联网校验
- 本轮不允许通过“导入更新包”做降级
- 本轮不改变 `state/` 永不覆盖的既有约束

## 校验规则

### 结构校验

更新包必须满足：

- 存在 `version.json`
- `version.json` 可以被解析为合法 JSON
- `version.json` 中存在非空 `version`
- 至少包含一项可更新分发内容

“可更新分发内容”继续沿用现有范围：

- `OpenClawLauncher.exe`
- `_internal`
- `runtime`
- `assets`
- `tools`
- `README.txt`
- `version.json`

如果除了 `version.json` 以外没有任何可替换分发内容，视为坏包或残包，直接拦截。

### 版本规则

- 必须同时读取当前便携包版本和更新包版本
- 只允许“更新包版本严格高于当前版本”
- 同版本包直接拦截，提示“当前已经是这个版本，无需导入”
- 旧版本包直接拦截，提示“这是旧版本，请改用恢复更新备份”

### 版本比较约定

当前项目版本样式至少包含：

- `v2026.04.1-dev`
- `v2026.04.2`

比较规则采用：

1. 先比较去掉前缀 `v` 后的数字段
2. 同数字段时，无后缀稳定版高于带后缀版本
3. 同数字段且都带后缀时，再按后缀字符串比较

这样可以满足当前产品最关键的用户场景：

- `v2026.04.2` > `v2026.04.1-dev`
- `v2026.04.1` > `v2026.04.1-dev`
- `v2026.04.1-dev` == `v2026.04.1-dev`

## 方案

### 服务边界

在 `launcher/services/local_update.py` 内补齐本地更新导入前的校验逻辑，保持更新和恢复相关逻辑集中。

建议抽出两个小单元：

- 更新包元数据读取/校验
- 版本字符串比较

`LocalUpdateImportService.import_package()` 在创建备份前先完成所有校验，只在确认包合法且版本更高后才进入复制流程。

### 错误文案

错误直接抛中文 `ValueError` / `FileNotFoundError`，沿用应用层现有错误边界直接展示，不再额外做技术化包装。

建议最小文案：

- `更新包缺少 version.json。`
- `更新包的 version.json 不是合法 JSON。`
- `更新包缺少有效的版本号。`
- `更新包中没有可导入的程序文件。请重新选择完整的便携包目录。`
- `当前已经是版本 {version}，无需重复导入。`
- `所选更新包版本 {incoming} 低于当前版本 {current}。如果要回退，请改用“恢复更新备份”。`

## 测试

- 服务测试：
  - 完整更新包可正常导入
  - 缺少 `version.json` 时失败
  - `version.json` 非法 JSON 时失败
  - 只有 `version.json` 没有分发内容时失败
  - 同版本包失败
  - 旧版本包失败并提示走恢复入口
  - `stable > dev` 的版本比较成立
- 非 UI 层回归：
  - 旧版本和同版本在进入复制前就被拦住，不创建或不使用备份目录进行替换

## 取舍

- 把“导入更新包”严格收敛为升级入口，比同时支持升级/降级更符合小白用户心智，也能减少误操作。
- 数字签名与哈希校验当然更强，但在当前阶段成本更高，且不解决“用户误选旧包/同版本包”这个更现实的问题；因此这轮优先把结构和版本规则收紧。
