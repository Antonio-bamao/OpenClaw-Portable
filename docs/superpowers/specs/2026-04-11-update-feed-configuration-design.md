# 真实更新源配置设计

## 背景

项目已经具备：

- 本地导入更新
- 更新回滚与恢复
- `update-manifest.json` 离线完整性校验
- 在线检查更新、下载 zip、解压到临时目录并交给本地导入链路

当前在线更新服务仍使用代码里的占位更新源地址，离“真正可发布”只差最后一层配置收口：正式更新源 URL 应该有一个内置默认值，同时允许在开发、灰度或临时切换场景下通过环境变量覆盖。

## 目标

为在线更新补齐可发布的更新源配置机制：

- 代码内保留一个默认正式更新源 URL
- 支持环境变量 `OPENCLAW_PORTABLE_UPDATE_FEED_URL` 覆盖
- 在线更新服务统一通过解析函数获取最终更新源地址

## 非目标

- 本轮不搭建真实静态站点
- 本轮不发布真实 `update.json`
- 本轮不增加 UI 设置项
- 本轮不把更新源地址写进 `state/` 配置

## 方案

### 配置优先级

按以下顺序解析更新源 URL：

1. 显式传入的请求级地址
2. 环境变量 `OPENCLAW_PORTABLE_UPDATE_FEED_URL`
3. 代码内默认正式地址

这样可以同时满足：

- 小白用户无需配置
- 本地联调可临时切到测试 `update.json`
- 未来正式地址调整有集中收口点

### 默认正式地址

本轮先在代码中集中定义默认正式地址常量，不再散落在业务实现里。

如果当前尚未提供真正上线地址，默认值可以暂时沿用现有占位地址，但解析逻辑、测试和文档都应围绕“正式默认值 + 环境变量覆盖”组织，后续只需替换常量即可。

### 模块边界

建议新增一个极小的解析模块，例如：

- `launcher/services/update_feed.py`

职责：

- 暴露默认正式地址常量
- 解析环境变量覆盖
- 校验解析出的 URL 非空

`OnlineUpdateService` 不直接关心环境变量细节，只接收解析结果。

## 测试

- 无环境变量时使用默认正式地址
- 有环境变量时优先使用环境变量
- 显式传入地址时优先级最高

## 取舍

- 不把更新源地址写进 `state/openclaw.json`
  - 对小白用户没价值
  - 会增加配置面和支持成本
- 先用环境变量覆盖，而不是 UI 设置项
  - 更适合开发、灰度和售后临时排障
  - 不会把高级配置暴露给普通用户
