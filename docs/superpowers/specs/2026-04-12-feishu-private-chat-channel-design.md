# 飞书私聊渠道 MVP 设计

## 背景

当前项目已经具备：

- 真实 OpenClaw runtime 接入
- 启动器主面板、首次引导、更新、诊断、恢复出厂闭环
- 便携路径与状态目录约束
- 真实便携包稳定性验证能力

但“渠道接入”仍停留在产品文档层，没有任何一个渠道形成真正可卖、可演示、可售后的闭环。继续同时铺微信、QQ、企微和飞书，只会把有限研发资源摊薄成四个半成品。因此本轮要把范围收窄成一个真正可交付的渠道 MVP。

在四个候选渠道里，飞书最适合作为第一刀：

- 官方应用与机器人链路清晰
- 应用凭据、鉴权与消息接口文档完整
- 合规与风控边界比微信/QQ 更明确
- 适合先做“启动器优先”的桌面交付体验

本轮设计将飞书渠道收敛成“私聊聊天闭环 MVP”。

## 目标

本轮只实现以下闭环：

1. 用户在启动器里完成飞书应用配置。
2. 启动器可测试飞书应用凭据有效性。
3. 用户可在启动器中启用飞书私聊渠道。
4. OpenClaw runtime 启动后建立飞书渠道连接。
5. 飞书用户私聊 Bot 发送消息后，消息能进入 OpenClaw，并将回复返回到同一飞书私聊会话。
6. 启动器能展示清晰的中文状态和错误，而不是底层异常。

## 非目标

本轮明确不做：

- 微信、QQ、企微接入
- 飞书群聊、@提及、群白名单
- WebUI 双入口配置
- 多飞书应用 / 多账号管理
- 自动修复飞书后台配置错误
- 飞书渠道的复杂权限矩阵
- 不依赖 OpenClaw 原生渠道能力而自建一套 Python 消息桥

## 用户场景

典型用户操作流如下：

1. 用户在启动器主界面进入“渠道管理”。
2. 选择“飞书”渠道卡片。
3. 输入 `App ID` 和 `App Secret`。
4. 点击“测试连接”。
5. 若凭据有效，启动器显示“测试通过，可启用飞书私聊”。
6. 用户点击“启用飞书私聊”。
7. 启动器将飞书配置落到便携状态目录，并把飞书渠道启用信息编排进 runtime 配置。
8. 用户启动 OpenClaw runtime。
9. 启动器显示飞书状态从“连接中”变为“已连接，可接收私聊”。
10. 飞书用户私聊 Bot，消息进入 OpenClaw，生成回复后再回到飞书私聊会话。

## 设计原则

- 启动器优先：所有关键配置、状态、错误提示都先在启动器完成。
- 私聊先行：先做最小可卖闭环，不引入群聊复杂度。
- 配置与状态分层：用户资产和运行态快照分离。
- 运行时复用：消息收发复用 OpenClaw runtime / 插件能力，不在 Python 侧重复造桥。
- 中文可理解：用户看到的是“配置无效”“飞书后台配置不完整”之类的状态，不是底层堆栈。

## 官方能力前提

本设计建立在飞书开放平台的官方能力之上：

- 应用级鉴权：通过应用凭据换取 `tenant_access_token`
- 应用机器人消息接口：用于回发消息
- 事件订阅 / 长连接接入：用于接收用户私聊消息

本轮不引入任何非官方个人号方案。

## 架构设计

### 1. 启动器层

启动器新增一个飞书渠道卡片或独立面板，至少提供：

- 输入框：`App ID`、`App Secret`
- 操作按钮：`测试连接`、`启用飞书私聊`、`停用`、`重试`
- 状态展示：当前状态、最近一次验证时间、最近一次错误
- 帮助入口：跳转离线帮助页，指导用户完成飞书开放平台后台配置

启动器不直接转发飞书消息，只负责：

- 配置读写
- 凭据测试
- 启用/停用编排
- 状态渲染
- 错误提示

### 2. 服务层

新增 `FeishuChannelService`，只承担以下职责：

- 读写 `state/channels/feishu/` 配置
- 校验字段完整性
- 通过飞书官方鉴权接口测试凭据有效性
- 将飞书启用配置编排进 runtime 所需配置
- 从 runtime 或日志中推断渠道运行状态
- 生成面向 UI 的中文状态文案

该服务不直接承担模型调用，不自行托管消息循环。

### 3. 运行时层

飞书私聊消息链路由 OpenClaw runtime 侧承担：

- 加载飞书渠道插件
- 建立飞书事件接收链路
- 将收到的私聊消息送入 OpenClaw 当前对话入口
- 将生成结果发送回飞书

这样可以让“配置编排”和“消息收发”边界清晰：

- 启动器负责让飞书渠道“可配置、可启用、可观测”
- runtime 负责让飞书渠道“真收消息、真回消息”

### 4. 目录结构

本轮沿用现有便携目录约束：

```text
state/
  channels/
    feishu/
      config.json
      status.json
assets/
  guide/
    setup-feishu.html
runtime/
  plugins/
    feishu/
```

## 数据模型

### 1. `state/channels/feishu/config.json`

用于保存用户资产级配置，建议字段：

```json
{
  "appId": "cli_xxx",
  "appSecret": "xxx",
  "enabled": true,
  "botAppName": "OpenClaw Bot",
  "lastValidatedAt": "2026-04-12T10:00:00Z"
}
```

说明：

- `appId` / `appSecret` 以明文保存在本地 `config.json`
- 这样做是为了保证启动器读写简单、可恢复、可诊断
- 诊断导出时必须脱敏

### 2. `state/channels/feishu/status.json`

用于保存运行态快照，建议字段：

```json
{
  "state": "connected",
  "lastError": "",
  "lastConnectedAt": "2026-04-12T10:02:00Z",
  "lastMessageAt": "2026-04-12T10:05:00Z"
}
```

说明：

- `status.json` 可被 runtime 或启动器刷新
- 其意义是“上一次已知状态”，不是绝对实时状态
- 若状态文件缺失，UI 应回退为“未配置”或“未知状态”，而不是报错

## 状态机

飞书渠道至少支持以下状态：

- `未配置`
- `配置无效`
- `待启用`
- `连接中`
- `已连接`
- `连接失败`
- `需重新配置`

状态语义：

- `未配置`：没有凭据或配置文件为空
- `配置无效`：凭据测试失败
- `待启用`：凭据通过，但未启用飞书渠道
- `连接中`：runtime 已启动，正在建立飞书链路
- `已连接`：飞书私聊链路已就绪
- `连接失败`：插件初始化或消息链路建立失败
- `需重新配置`：凭据失效、飞书后台变更、设备迁移等导致不能继续使用

## 运行链路

### 1. 测试连接

“测试连接”只做两层验证：

#### 凭据层验证

- 使用 `App ID` / `App Secret`
- 调飞书官方鉴权接口换取访问令牌
- 若成功，说明应用凭据有效

#### 运行层前置验证

- 校验本地配置是否满足飞书渠道启用所需字段
- 不在此处伪造聊天消息
- 真正的聊天链路以渠道启用后的 runtime 状态为准

这样可以把“凭据错误”和“运行链路错误”分开，减少误判。

### 2. 启用飞书私聊

点击“启用飞书私聊”后：

1. 启动器确保本地 `config.json` 已保存并通过凭据测试。
2. 启动器更新飞书渠道启用标志。
3. 启动器将飞书渠道配置编排进 runtime 所需配置。
4. 若 runtime 已运行，则提示用户重启或自动触发安全重启。

### 3. 私聊聊天闭环

正式聊天链路为：

1. 飞书用户私聊 Bot。
2. 飞书事件进入 runtime 的飞书插件。
3. 插件将消息送入 OpenClaw 对话入口。
4. OpenClaw 生成回复。
5. 插件将回复回发到原私聊会话。
6. `status.json` 记录最近成功收发时间。

## 帮助与引导

本轮必须新增飞书离线帮助页 `assets/guide/setup-feishu.html`，至少覆盖：

- 如何在飞书开放平台创建应用
- 如何获取 `App ID` / `App Secret`
- 需要勾选或配置的机器人/事件订阅项
- 常见错误说明
- 风险边界说明：本轮只支持私聊，不支持群聊

帮助页应从启动器飞书卡片直接打开。

## 错误处理

本轮只做 5 类高频错误的清晰映射：

### 1. 凭据错误

- 表现：获取飞书访问令牌失败
- UI 提示：`配置无效，请检查 App ID / App Secret`

### 2. 飞书后台配置不完整

- 表现：凭据正确，但事件订阅或机器人权限链路不完整
- UI 提示：`飞书应用配置不完整，请按帮助页完成开放平台设置`

### 3. runtime 未启动

- 表现：用户配置了飞书，但尚未启动 OpenClaw
- UI 提示：`请先启动服务，再启用飞书私聊`

### 4. 渠道连接失败

- 表现：插件加载失败、长连接失败、回消息失败
- UI 提示：`飞书连接失败，可查看诊断并重试`

### 5. 需重新配置

- 表现：凭据失效、设备迁移或飞书侧变更导致链路不可用
- UI 提示：`当前飞书渠道需重新配置`

## 诊断与脱敏

飞书渠道纳入现有诊断体系时应满足：

- 诊断包可以包含飞书渠道状态
- `appSecret` 必须脱敏
- token 若出现在状态或日志中，只保留前缀
- 最近错误可导出
- 最近成功连接时间和最近消息时间可导出

## 测试策略

### 1. 单元测试

至少覆盖：

- 飞书配置读写
- 凭据缺失/无效/有效状态映射
- 状态机转换
- 渠道启用编排
- 诊断导出脱敏

### 2. 运行集成测试

至少覆盖：

- runtime 启动后能识别飞书渠道启用状态
- 渠道启用后状态从“待启用”到“连接中”再到“已连接”的路径
- 插件未就绪时能稳定落入“连接失败”

### 3. 手工验收

需要真实飞书开发环境做一次端到端验证：

1. 新建飞书应用
2. 填入 `App ID / App Secret`
3. 测试连接成功
4. 启用飞书私聊
5. 启动 OpenClaw
6. 在飞书中私聊 Bot
7. 收到 OpenClaw 回复

## 验收标准

本轮完成后，以下 6 条必须全部成立：

1. 启动器中存在飞书渠道卡片或飞书配置面板。
2. 用户能保存 `App ID / App Secret`。
3. 用户能在启动器中完成飞书凭据测试。
4. 用户能在启动器中启用飞书私聊渠道。
5. 飞书用户私聊 Bot 后，能收到 OpenClaw 回复。
6. 失败时用户看到的是中文可执行提示，而不是裸异常。

## 未来扩展

本设计刻意为后续能力预留空间，但不在本轮实现：

- 飞书群聊与群白名单
- WebUI 渠道管理页
- 企微 / 微信 / QQ 渠道并列抽象
- 多渠道统一状态面板
- 多飞书应用管理

后续实现这些能力时，应复用本轮形成的三层结构：

- 启动器编排层
- 渠道配置/状态存储层
- runtime / 插件消息链路层
