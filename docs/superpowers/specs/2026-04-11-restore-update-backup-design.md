# 从更新备份恢复旧版本设计

## 背景

项目已经具备“导入本地更新包 -> 自动备份旧分发内容 -> 复制失败自动回滚”的最小闭环，但用户暂时还无法在非命令行场景下主动恢复到某一次更新前的旧版本。当前 `.context` 已把“从备份恢复旧版本”列为更新/回滚闭环的优先补强项。

## 目标

新增一个面向主界面的“从更新备份恢复旧版本”入口，让用户可以从 `state/backups/updates/` 中选择一份历史备份，将当前便携包的分发内容恢复到那次更新前的状态。

## 非目标

- 本轮不做在线更新。
- 本轮不做签名校验或更复杂的版本合法性校验。
- 本轮不恢复或覆盖 `state/` 下的配置、缓存、诊断包或 workspace。
- 本轮不做“多版本历史列表 UI”，先采用目录选择器。

## 约束

- 恢复范围必须与本地导入更新包一致，仅覆盖：
  - `OpenClawLauncher.exe`
  - `_internal/`
  - `runtime/`
  - `assets/`
  - `tools/`
  - `README.txt`
  - `version.json`
- 恢复前必须先把“当前分发内容”再次备份到 `state/backups/updates/`，这样恢复失败时仍然可以回滚。
- 任何恢复流程都不得覆盖 `state/`。
- 恢复前应先停止 runtime，恢复后重置 controller 的 prepared 状态。

## 方案

### 用户流程

1. 用户在主界面点击“恢复更新备份”。
2. 应用先弹出确认框，明确提示：
   - 只恢复程序分发内容，不覆盖 `state/`
   - 当前版本会先自动备份
   - 完成后需要重启启动器
3. 用户确认后，选择 `state/backups/updates/` 下的一份备份目录。
4. 应用调用恢复服务执行恢复。
5. 成功时弹出提示，展示恢复来源目录名或恢复后的版本号，并提醒重启。
6. 失败时走现有错误边界，展示中文错误提示。

### 服务边界

- 新增 `RestoreUpdateBackupService`，职责与 `LocalUpdateImportService` 对称：
  - 校验备份目录存在且不是当前便携包根目录
  - 从备份目录读取可恢复的分发条目
  - 在恢复前再次备份当前分发内容
  - 覆盖目标目录
  - 中途失败时自动回滚到恢复前状态
- `LauncherController` 新增 `restore_update_backup()`，负责：
  - 停止 runtime
  - 调用恢复服务
  - 将 `_prepared` 置回 `False`
- `OpenClawLauncherApplication` 新增恢复入口处理函数和目录选择器/确认框。
- 主界面新增一个次级按钮，但仍保持现有按钮组织方式。

### 结果对象

恢复服务返回与本地更新导入相似的结果对象，至少包含：

- `restored_version`
- `backup_dir`
- `restored_entries`
- `source_backup_dir`

如果无法读取 `version.json`，允许 `restored_version` 为空字符串，UI 使用目录名兜底。

## 错误处理

- 备份目录不存在：抛 `FileNotFoundError`。
- 备份目录与当前便携包根目录相同：抛 `ValueError`。
- 备份目录缺少任何可恢复分发条目：抛 `FileNotFoundError`，避免“恢复成功但什么都没发生”。
- 覆盖途中失败：必须自动回滚到恢复前状态。

## 测试

- 服务测试：
  - 可从一份更新备份恢复旧分发内容，同时保留 `state/`
  - 恢复过程中复制失败时，会回滚到恢复前的当前版本
  - 空备份目录会报错
- Controller 测试：
  - 恢复前会先 stop runtime，恢复后 `_prepared=False`
- App/UI 测试：
  - 点击恢复入口后会先确认，再选择目录，再调用 controller
  - 成功后显示“已恢复旧版本，请重启启动器”类提示
  - 主界面展示新的次级按钮文本

## 取舍

- 目录选择器比“内建历史列表”更朴素，但实现成本更低，且完全复用现有便携目录结构。
- 恢复范围继续锁定在分发内容，能够最大化保护用户配置和工作资料，符合当前售后安全优先的产品定位。
