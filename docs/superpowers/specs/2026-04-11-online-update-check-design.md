# 在线检查更新设计

## 背景

项目已经具备以下本地更新能力：

- 手动导入更新包
- 导入前结构校验、版本校验
- `update-manifest.json` 离线 `SHA-256` 完整性校验
- 更新前自动备份
- 替换失败自动回滚
- 从更新备份恢复旧版本

当前仍缺少一个面向小白用户的“联网发现新版本并下载”的入口。产品文档已将“检查更新”列为 P1 能力，但如果本轮直接实现“联网下载后自动替换”，会把网络、下载、解压、文件占用、替换、回滚和 UI 状态一次性叠在一起，风险过高。

## 目标

本轮实现一个保守但可用的在线更新闭环：

- 从固定静态 JSON 地址检查最新版本
- 向用户展示更新版本和更新说明
- 下载更新 zip 到本地临时目录
- 解压为便携包目录
- 将解压目录交给现有本地导入链路处理

## 非目标

- 本轮不实现后台静默自动更新
- 本轮不直接在下载完成后绕过本地导入链路做原地替换
- 本轮不做数字签名
- 本轮不做差分更新
- 本轮不设计复杂的更新进度 UI

## 用户流程

1. 用户点击主界面“检查更新”。
2. 应用请求固定的 `update.json` 地址。
3. 如果当前已是最新版本：
   - 弹窗提示“当前已经是最新版本”。
4. 如果发现新版本：
   - 弹窗展示最新版本号和更新说明。
   - 用户确认后开始下载 zip。
5. 应用把 zip 下载到 `%TEMP%\OpenClawPortable\updates\downloads\`。
6. 应用把 zip 解压到 `%TEMP%\OpenClawPortable\updates\packages\<version>\`。
7. 应用把解压目录交给现有 `import_update_package()`。
8. 导入成功后提示“已更新到 xxx，请重启启动器完成切换”。
9. 任一步失败，立即中止并展示中文错误提示，不继续后续动作。

## 远端元数据格式

更新源使用固定静态 JSON 地址，结构最小化为：

```json
{
  "version": "v2026.04.2",
  "notes": [
    "升级到新的 OpenClaw runtime",
    "修复本地更新回滚问题"
  ],
  "packageUrl": "https://example.com/OpenClaw-Portable-v2026.04.2.zip"
}
```

### 字段规则

- `version`
  - 必填
  - 必须是合法版本号
- `notes`
  - 可选
  - 缺省时按空列表处理
- `packageUrl`
  - 必填
  - 必须是非空字符串

本轮不在 `update.json` 中重复放哈希清单，因为 zip 解压后的目录仍然会走现有 `version.json + update-manifest.json` 校验链路。

## 架构设计

### 1. `UpdateCheckService`

职责：

- 请求固定更新源 URL
- 解析 `update.json`
- 校验字段结构
- 比较“当前版本”和“远端版本”
- 返回以下三类结果之一：
  - 已是最新
  - 有新版本
  - 请求或解析失败

边界：

- 不负责下载 zip
- 不负责 UI
- 不负责替换文件

### 2. `UpdateDownloadService`

职责：

- 下载 zip 到临时目录
- 解压 zip
- 找到实际便携包根目录
- 返回可交给本地导入链路的包目录

边界：

- 不做文件替换
- 不做版本比较
- 不跳过本地导入校验

### 3. 复用 `LocalUpdateImportService`

职责不变：

- 版本校验
- `update-manifest.json` 校验
- 备份
- 替换
- 回滚

在线更新永远不直接改写分发内容，而是把下载并解压后的目录交给现有导入链路。

## 临时目录约定

建议统一放到：

- `%TEMP%\OpenClawPortable\updates\downloads\`
- `%TEMP%\OpenClawPortable\updates\packages\`

下载文件和解压目录都不放进便携包根目录，避免污染用户 U 盘内容和状态目录。

## UI 设计

主界面新增一个次级按钮：`检查更新`

交互原则：

- 不做复杂进度条，先做同步型“检查中 / 下载中”短提示
- 有更新时必须先给用户看版本号和更新说明，再让用户确认
- 成功后明确提示需要重启启动器
- 失败时直接给中文错误提示，不进入半成功状态

## 错误处理

### 检查阶段

- 更新源不可达或超时
  - `当前无法连接更新服务器，请稍后再试。`
- `update.json` 非法 JSON 或字段缺失
  - `更新信息格式错误，请联系售后。`

### 下载阶段

- 下载失败
  - `更新包下载失败，请检查网络后重试。`
- zip 解压失败
  - `更新包解压失败，可能是下载不完整。`
- zip 内没有可识别的便携包根目录
  - `更新包内容不完整，请重新下载。`

### 导入阶段

导入失败不另起一套新逻辑，直接复用现有本地导入错误：

- 版本不合法
- 版本不够新
- `update-manifest.json` 缺失或损坏
- 关键文件哈希不匹配
- 替换失败
- 自动回滚失败

## 测试设计

### `UpdateCheckService`

- 返回“已是最新”
- 返回“发现新版本”
- `update.json` 非法 JSON 时失败
- 缺少 `version` 或 `packageUrl` 时失败

### `UpdateDownloadService`

- 能下载并解压一个本地测试 zip
- 下载失败时失败
- zip 损坏时失败
- zip 解压后能定位便携包根目录

### 应用层 / 控制器编排

- “检查更新”无更新时显示提示
- 有更新时先确认，再下载，再调用现有导入链路
- 成功后显示“请重启启动器”
- 检查、下载、解压任一步失败都走统一错误边界

## 实施策略

本轮优先把“检查更新 + 下载到本地 + 交给现有导入链路”做通，不在当前范围里扩展静默更新、后台线程或差分包。这样可以最大化复用已经验证过的本地更新安全链路，减少新引入的状态面。

## 取舍

- 选择静态 JSON 地址，而不是直接对接 GitHub Release：
  - 更容易控制格式
  - 更适合后续自有发布链路
  - 减少 GitHub API/网络波动带来的不确定性
- 选择“下载后交给现有导入链路”，而不是“自动直接替换”：
  - 风险更低
  - 复用现有回滚能力
  - 更适合当前 MVP 阶段
